## 一、核心设计原则

### 1. 让孩子上瘾的心理学机制

| 机制 | 实现方式 | 参考案例 |
|------|---------|---------|
| **收集欲** | 每颗诞生的星球是一张"收藏卡"，可以回顾、展示 | Pokemon 卡牌 |
| **即时反馈** | 任何操作都有视觉变化（闪光、抖动、粒子） | Duolingo 的 streak |
| **渐进式解锁** | 不是一开始全给，而是随着行为逐步解锁新能力 | 游戏技能树 |
| **个性化** | 星球命名、emoji、颜色自选，"我的星球我做主" | Animal Crossing |
| **社交展示** | 家长朋友圈/家庭群可以分享星球证书 | 支付宝年度账单 |

### 2. "不用3D"的视觉方案

> 核心判断：Three.js 成本高、开发慢、小程序端兼容性差。用 2D + 动画组合拳效果一样好。

**方案：SVG + CSS Animation + Lottie**

```
星球 = 多层 SVG 叠加

Layer 0: 基底圆（颜色自选）
Layer 1: 大气层光晕（CSS pulse 动画）
Layer 2: 地形纹理（完成知识卡后叠加）
Layer 3: 生命元素（完成里程碑后出现小动物/植物）
Layer 4: 特效层（闪光粒子用 Lottie 预渲染动画）

变暗 = 逐层降低 opacity + 灰色滤镜 + 乌云 SVG
```

**Lottie 资源：**
- 星球诞生动画（预渲染 JSON）
- 里程碑完成庆祝动画
- 星球旋转（CSS rotate 即可）
- 优点：设计师一次出图，多端复用（小程序/Web 都能用）

### 3. 孩子 vs 家长的差异化体验

| 维度 | 孩子端 | 家长端 |
|------|--------|--------|
| 语言 | 拟人化（"你的星球在等你哦"） | 数据化（"宝贝已完成 3/5 里程碑"） |
| 交互 | 大按钮、少文字、多动画 | 列表、筛选、批量操作 |
| 推送 | 趣味化提醒 | 进度报告、教育建议 |
| 色彩 | 鲜艳、饱和 | 柔和、专业 |

## 二、子系统架构设计

### 事件驱动，解耦现有系统

```
┌─────────────────────────────────────────────────┐
│              Planet Service (新)                  │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐   │
│  │ Planet   │  │Card Gen  │  │ Milestone     │   │
│  │ Manager  │  │ AI 服务  │  │ Engine        │   │
│  └────┬─────┘  └────┬─────┘  └──────┬────────┘   │
│       │              │               │            │
│       └──────────────┼───────────────┘            │
│                      │                            │
│              Event Bus (Redis Stream / RabbitMQ)   │
└──────────────────────┼────────────────────────────┘
                       │ 订阅事件
         ┌─────────────┼─────────────┐
         ▼             ▼             ▼
   ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ Items    │  │ Coins    │  │ Exams    │
   │ Service  │  │ Service  │  │ Service  │
   └──────────┘  └──────────┘  └──────────┘

   发布事件示例：
   - task.completed      → 习惯里程碑推进
   - coins.updated       → 攒币里程碑推进
   - exam.passed         → 知识考试里程碑完成
   - card.read           → 知识卡片已读
```

**关键设计：**
- 星球模块不直接读其他模块的数据库
- 通过事件总线通信，松耦合
- 如果某个服务暂时不可用，事件可以重试
- 方便后续拆分为独立微服务

## 三、功能融合方案

### 与现有模块的融合点

```
星球 = 所有现有功能的"容器"和"叙事层"

WishPlanet
  ├── 知识卡片 → AI 生成，孩子阅读 → 星球变亮
  ├── Items    → 习惯打卡任务      → 星球长出水/绿色纹理
  ├── Coins    → 学习币攒目标      → 星球长出建筑/装饰
  ├── Exams    → 守护者考试        → 星球闪光 ✨
  ├── Achievements → 星球诞生解锁   → 专属成就徽章
  └── Prizes   → 实物关联          → 星球诞生后解锁购买
```

### 融合规则

1. **星球是主叙事，现有功能是工具**
   - 用户看到的是"我在让星球成长"，不是"我在完成任务"
   - 现有功能被"包裹"在星球体验里

2. **数据流向：双向但单向为主**
   - 星球模块 → 消费其他模块的事件
   - 其他模块 → 不需要感知星球的存在
   - 唯一反向：星球诞生后触发成就解锁

3. **渐进式上线**
   - MVP 可以只用知识卡片（不依赖其他模块）
   - Phase 2 再接入 Items/Coins/Exams
   - 每阶段都可以独立上线

## 四、如何保证孩子喜欢 — 验证方法

### 低成本验证路径

**1. 纸面原型测试（1周内）**
- 用 Figma 做出星球生长的高保真原型
- 找 3-5 个目标年龄段的孩子试用
- 观察他们是否主动想"再看一次""再点一次"

**2. A/B 两个视觉方向**
- 方向 A：卡通星球（可爱路线）
- 方向 B：写实星球（科幻路线）
- 让目标用户选，数据说话

**3. "最小可爱产品"原则**
- 不追求 3D/完美动画
- 追求 3 个关键时刻的极致体验：
  - **第一次看到自己的星球** → 哇！
  - **完成第一个里程碑** → 好酷！
  - **星球诞生那一刻** → 我要给妈妈看！

**4. 家长反馈循环**
- 星球详情页加一个"宝贝喜欢吗？"简单投票
- 收集真实用户反馈，迭代设计

## 五、小程序架构方案 (Taro + React)

### 为什么 Taro

- 一套代码编译到微信/支付宝/字节等多端
- React 技术栈，团队可以直接上手
- 生态成熟，组件库丰富
- 与现有 Web 端可以共享部分逻辑（Hooks、状态管理）

### 技术栈选型

| 层级 | 选择 | 理由 |
|------|------|------|
| 框架 | Taro 4 + React 18 | 多端兼容，团队熟悉 |
| 状态管理 | Zustand | 轻量，比 Redux 简单 |
| UI 组件 | Taroify / NutUI | 移动端组件库，小程序适配好 |
| 样式 | SCSS + CSS Modules | scoped，避免冲突 |
| 动画 | Lottie-miniapp + CSS | 轻量动画方案 |
| 网络请求 | Taro.request 封装 | 统一拦截器 |
| 类型 | TypeScript | 与现有项目一致 |
| 构建 | Taro CLI | 官方工具 |

### 与后端的接口

```
小程序 ←→ 现有 FastAPI 后端（同一套 API）
不需要为小程序单独开发后端

认证流程：
1. 微信授权登录 → 获取 code
2. 后端用 code 换 openId（微信 API）
3. 匹配/创建用户 → 返回 JWT
4. 后续请求带 JWT

注意：如果后端已有 Casdoor，可以
- Casdoor 支持微信扫码登录
- 或者加一层微信 OAuth 适配
```

### 多端适配策略

```
优先：微信小程序（最大用户群）
其次：抖音小程序（流量红利）
最后：支付宝小程序（如果需要）

Taro 的差异化：
- 用 process.env.TARO_ENV 判断平台
- 平台特有代码放在 components/xxx.wx.tsx / xxx.tt.tsx
- 核心逻辑共用，UI 层按需适配
```
